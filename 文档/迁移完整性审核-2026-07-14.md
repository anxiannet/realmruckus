## 四、关键缺失项

## P0：阻止新仓库成为真正主仓库的问题

### 1. 旧模拟器主体尚未迁移

旧仓库仍拥有完整或接近完整的：

- 游戏循环
- AI行动选择
- 攻击方案评分
- 战斗顺序与目标枚举
- 扩展包行为
- 逐卡触发统计
- 批量运行器
- 命令行入口

新仓库目前只有基础引擎骨架，不能运行正式完整对局。

### 2. 新战斗模块没有迁移“战斗方案枚举”

旧模拟器会枚举：

- 进攻单位顺序
- 每次攻击选择的防守目标
- 不同结果组合

AI再从结果中选择方案。

新引擎当前只按传入顺序攻击第一个存活防守单位，行为不等价。

这会直接改变：

- 进攻成功率
- AI决策质量
- 平均回合
- 区域回流
- 历史结果可复现性

### 3. 卡牌实例模型不完整

新引擎的手牌、牌库和弃牌堆目前使用 `Unit`。

但公共牌库同时包含：

- 单位牌
- 效果牌

因此当前模型无法完整表达效果牌实例、卡牌UID、牌库洗牌、弃牌和取消响应。

需要建立通用 `CardInstance`，再区分：

```text
UnitCardInstance
EffectCardInstance
```

或使用统一卡牌实例加 `kind` 字段。

### 4. 尚有13类效果未实现

包括：

- 查看随机手牌
- 社交交换
- 弃牌堆回收
- 查看牌库顶并选择
- 随机拿取手牌
- 暗选交换
- 被击败替代结算
- 单位转移至己方区域
- 额外部署
- 立即进攻
- 社交获得驻军
- 条件抽牌
- 支付弃牌后返回单位

### 5. 完整回合执行器未建立

当前只有阶段枚举，没有把以下步骤串成可运行对局：

```text
抽牌
→ 行动选择
→ 支付费用
→ 响应窗口
→ 效果结算
→ 区域回流
→ 即时胜利
→ 手牌上限
→ 牌库耗尽判定
→ 下一玩家
```

### 6. AI策略未迁移

缺少：

- `human_like`
- `stress_attack`
- 区域需求评分
- 卡牌优先级
- 防守保留策略
- 进攻计划评分
- 社交回应概率
- 扩展牌触发决策

因此新仓库目前不能复现历史平衡数据。

## P1：测试与结果缺失

### 7. 原始测试结果未迁移

目前只保存了Markdown汇总。

尚未迁移：

- 每组3000局JSON
- 压力测试JSON
- 构建产物
- 逐卡触发统计
- 逐卡使用后胜率相关性
- 每局原始结果或可追溯摘要

这使得历史结果无法自动复算和比较。

### 8. 正式批量运行器未建立

目标入口仍停留在文档：

```text
python -m simulation.run ...
```

实际 `simulation.run` 尚不存在。

### 9. 正式模拟CI未迁移

当前CI只进行：

- Python编译
- 单元测试
- JSON结构校验

缺少：

- 2、3、4人烟雾对局
- 100局回归
- 3000局正式测试
- JSON结果上传
- 与历史基线的容差比较

### 10. CI在线通过状态未确认

仓库存在工作流文件，但当前审核没有取得明确的成功运行记录。

不能将“工作流文件存在”视为“CI已通过”。

## P1：唯一来源与主从关系问题

### 11. 原仓库仍保留冲突的权威声明

虽然迁移索引已经宣布新仓库为通用规则主仓库，但原仓库仍有大量文件继续声明：

```text
原仓库核心规则为唯一来源
原仓库基础卡表为唯一来源
```

这些文件尚未统一标记为：

```text
历史来源
具体表达
迁移核对文件
```

### 12. Realm Ruckus专属文件仍引用旧路径

例如主题方案仍引用：

```text
规范流程/换皮与卡号规范.md
```

但该路径不在新仓库中。

英文核心规则仍声明：

```text
规则/V1.3-核心规则.md
中文源规则优先
```

这与“新仓库通用规则为最高来源”的新结构冲突。

需要改为引用：

```text
文档/通用规则/核心机制.md
文档/通用规则/通用标准版卡表.md
```

### 13. 原仓库项目状态存在40张与42张口径冲突

原仓库历史状态文件记录过：

```text
42张标准版为确定方案
42+18=60张
```

但最新卡表、扩展卡表和正式模拟器使用：

```text
40张标准版
40+10=50张
```

新仓库采用了40+10方案，但尚未建立正式“冲突裁决与版本继承记录”。

必须明确：

- 42/60属于历史候选；
- 40/50属于当前迁移基线；
- 依据的具体源文件和提交；
- 为什么新主仓库采用40/50。

### 16. 数学模型仍为说明性文档

已有公式与指标，但缺少可执行模块：


缺少：

- 置信区间
- 不同版本差异检测
- 容差阈值
- 玩家位置偏差
- P95或更高长尾
- 自动基线回归报告

## 五、现有实现风险

### 1. 取消效果语义不完整

当前 `CANCEL_EFFECT` 只修改当前 `EffectContext.cancelled`。

正式实现需要：

- 指向被响应的效果实例；
- 管理响应窗口；
- 区分取消牌本身与被取消效果；
- 正确处理费用和弃牌去向。

### 2. 区域保护持续时间未实现

JSON写有持续时间，但引擎尚未在正确回合节点清除保护。

### 3. 替代击败效果未实现

目前普通击败直接进入弃牌堆，未处理特定机制的“改为洗回牌库”。

### 4. 效果目标验证尚不完整

当前部分执行器依赖调用者提前提供合法目标。

尚缺统一的：

- 目标枚举
- 精确数量检查
- 无合法目标判定
- 自己/对手限制
- 受保护区域限制
- 支付费用验证

### 5. 规则文档与JSON可能发生漂移

Markdown卡表与JSON目前是两份人工维护数据。

需要确定机器可读JSON是否成为机制卡表唯一数据源，Markdown由JSON生成；否则会形成新的平行卡表风险。

## 六、迁移完整的判定标准

只有同时满足以下条件，才能宣布迁移完成：

1. 全部标准版与扩展效果由 `effect_id` 实现。
2. 完整对局可在新仓库运行。
3. 战斗方案枚举与旧模拟器等价。
4. `human_like` 与 `stress_attack` 已迁移。
5. 2、3、4人正式模拟可运行。
6. 历史3000局结果在合理容差内复现。
7. 原始JSON结果已归档。
8. QDP与RRK均完成mechanic_id映射。
9. 旧仓库的权威声明完成清理或归档标记。
10. Realm Ruckus专属文件全部改为引用通用规则层。
11. 通用换皮、卡号、版本和生产流程已建立。
12. CI取得明确成功记录。
13. Markdown与JSON卡表不存在双重维护风险。

## 七、建议执行顺序

### 第一阶段：修正数据与权威关系

1. 确定JSON为通用机制卡表数据唯一来源。
2. Markdown卡表改为由JSON生成或只作解释。
3. 建立40/50对42/60的正式版本裁决记录。
4. 修正Realm Ruckus文件中的旧路径和旧权威声明。

### 第二阶段：迁移功能

1. 建立统一卡牌实例模型。
2. 迁移战斗方案枚举。
3. 补齐全部效果执行器。
4. 建立完整回合执行器。
5. 迁移两套AI策略。

### 第三阶段：迁移测试

1. 迁移批量运行器和CLI。
2. 迁移原始JSON结果。
3. 建立100局回归和3000局正式工作流。
4. 建立自动基线比较。

### 第四阶段：具体表达收束

1. 建立QDP映射。
2. 建立RRK映射。
3. 清理两个表达中的平行核心规则。
4. 将旧模拟器和旧规则标记为历史来源。

## 八、最终判断

当前新仓库已经具备：

```text
通用规则主仓库的结构基础
```

但尚不具备：

```text
完整替代旧仓库规则引擎与正式模拟系统的能力
```

因此当前准确状态应为：

```text
迁移进行中
结构层通过
功能层未通过
测试复现未通过
唯一来源切换未通过
```
